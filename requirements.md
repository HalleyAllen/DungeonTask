# DungeonTask 需求文档

> 最后更新：2026-05-24

## 项目概述

DungeonTask 是一款面向鸿蒙（HarmonyOS）平台的日常记录与自我管理应用，核心功能包括：任务清单、健身计划、句子收集、词语收集、目标/愿望清单、备忘录六个模块。

---

## 功能需求

### 1. 任务清单（每日工作记录）

- 支持按日期记录每日工作任务
- 任务支持：添加、编辑、删除、标记完成
- 任务列表按日期分组展示（今日 / 历史）
- 支持任务完成状态打卡
- **数据本地持久化存储**
- **支持数据导出**

### 2. 健身计划

- 支持设置健身计划规律（如：每周一/三/五练胸、背、腿等）
- 计划按周期排布，支持自定义训练项目
- 每次训练完成后可打卡标记
- 展示健身计划日历 / 列表视图
- 可查看历史打卡记录
- **数据本地持久化存储**
- **支持数据导出**

### 3. 句子收集（有意思的句子）

- 支持添加、编辑、删除句子
- 句子列表展示，支持快速浏览
- 支持按标签或分类查看（可选）
- **数据本地持久化存储**
- **支持数据导出**

### 4. 词语收集

- 支持添加、编辑、删除词语
- 每个词语记录：词语名称、拼音、释义
- 词语列表展示，支持快速浏览
- **数据本地持久化存储**
- **支持数据导出**

### 5. 目标 / 愿望清单

- 支持设置短期 / 长期目标
- 每个目标支持标记完成进度（百分比或步骤）
- 支持添加、编辑、删除目标
- 目标列表展示，已完成/进行中分类查看
- **数据本地持久化存储**
- **支持数据导出**

### 6. 备忘录 / 随手记

- 快速记录临时想法，不归类到具体模块
- 支持添加、编辑、删除备忘录
- 备忘录列表按时间倒序展示（最新的在最前）
- 支持按关键词搜索备忘录内容
- **数据本地持久化存储**
- **支持数据导出**

### 7. 数据统计 / 回顾（可选，后续迭代）

- 本周完成任务数统计
- 健身打卡天数统计
- 目标完成进度总览
- 历史句子 / 词语浏览记录

### 8. 多端数据同步（核心功能）

通过自部署的**同步服务器**实现多设备数据同步。用户在任意设备上产生密钥，其他端凭同一密钥连接服务器，自动拉取/推送增量数据。

#### 8.1 总体方案

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 手机 A   │    │ 平板 B   │    │ 手机 C   │
│ (创建密钥)│    │ (输入密钥)│    │ (输入密钥)│
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │               │               │
     │   HTTPS + HMAC-SHA256 签名    │
     │               │               │
     └───────┬───────┴───────┬───────┘
             │   RESTful API  │
             └───────┬───────┘
                     │
            ┌────────┴────────┐
            │  同步服务器       │
            │  (自部署 VPS)     │
            │  Go + SQLite     │
            └─────────────────┘
```

**核心理念**：一个**同步密钥（Sync Key）** = 一个数据空间。用户在第一台设备生成密钥，后续设备输入同一密钥即可加入同步组。

#### 8.2 技术选型（最优组合）

| 技术层 | 选型 | 理由 |
|--------|------|------|
| **服务端语言** | **Go** | 编成单二进制文件，无运行时依赖，内存占用 ~10MB，跨平台部署 |
| **服务端数据库** | **SQLite**（WAL 模式） | 零配置、零维护，与客户端表结构完全一致，天然兼容导出格式 |
| **服务端框架** | **Gin**（Go HTTP） | 高性能路由，中间件生态完善，支持请求限流 |
| **部署方式** | **Docker** 单容器 | 一键部署到任意 VPS / 树莓派 / NAS |
| **通信协议** | **HTTPS + JSON** | 通用性最强，调试方便，客户端 `@ohos.net.http` 原生支持 |
| **认证方式** | **HMAC-SHA256 签名** | 密钥不通过请求体传输，防重放攻击（Timestamp + Nonce） |
| **同步策略** | **Pull-Then-Push 增量同步** | 客户端先拉服务端最新数据合并，再推本地变更 |
| **冲突解决** | **Last-Write-Wins（LWW）** | 基于 `updateTime`，简单可靠，符合个人使用场景 |
| **部署平台** | 阿里云 / 腾讯云轻量服务器 | 最低配置 1C1G 即可（年费 ~¥100） |

> **为什么选择 Go 而不是 Node.js / Python？**
> - Go 编译为单二进制文件，不需要安装 Node/Python 运行时
> - 内存占用远低于 Node.js（~10MB vs ~50MB），适合低配 VPS
> - 并发模型（goroutine）天然适合处理多客户端同时同步
> - 交叉编译直接产出 Linux ARM/AMD64 二进制，树莓派也能跑

> **为什么选 SQLite 而不是 PostgreSQL/MySQL？**
> - 数据量极小（个人管理 App），SQLite 完全够用（单文件 ≤ 100MB）
> - 无需额外安装数据库服务，一个 Docker 容器解决一切
> - WAL 模式下读写并发性能足够支撑数十个客户端同时同步
> - 与客户端 SQLite 表结构一致，导出/备份直接复制文件即可

#### 8.3 同步密钥设计

```
密钥格式: dt_sk_<32位随机Hex>
示例:     dt_sk_a1b2c3d4e5f6...a1b2c3d4e5f6

生成方式: cryptoFramework.createRandom(32).toString('hex')
展示形式: 二维码 + 文本，方便扫码输入
```

| 密钥用途 | 说明 |
|----------|------|
| 设备 A 生成密钥 | 首次使用同步功能，生成密钥并注册到服务器 |
| 设备 B/C 加入 | 输入同一密钥，服务器验证后加入同步组 |
| 密钥重置 | 可在任一设备重新生成，旧密钥失效 |

#### 8.4 API 设计

所有请求需携带认证头 `X-Sync-Key` + HMAC 签名：

```
Base URL: https://<your-server>:8443/api/v1

┌────────┬─────────┬──────────────────────────────────────────┐
│ 方法    │ 路径     │ 说明                                     │
├────────┼─────────┼──────────────────────────────────────────┤
│ POST   │ /init   │ 注册新同步密钥（设备 A 首次创建）            │
│ POST   │ /verify │ 验证密钥有效性（设备 B/C 加入前校验）         │
│ POST   │ /pull   │ 拉取服务端最新数据（传入本地各表最后更新时间）   │
│ POST   │ /push   │ 推送本地变更数据到服务端                     │
│ DELETE │ /reset  │ 重置同步空间（清空服务端所有数据）             │
└────────┴─────────┴──────────────────────────────────────────┘
```

**Pull 请求体**（增量拉取）：
```json
{
  "tables": {
    "task":        { "lastSync": 1716500000000 },
    "collection":  { "lastSync": 1716480000000 },
    "memo":        { "lastSync": 0 },
    "goal":        { "lastSync": 1716500000000 },
    "collection_type": { "lastSync": 0 },
    "habit":       { "lastSync": 0 },
    "habit_log":   { "lastSync": 0 }
  }
}
```

**Pull 响应**（仅返回 `updateTime > lastSync` 的记录）：
```json
{
  "serverTime": 1716550000000,
  "changes": {
    "task": [
      { "id": 42, "title": "买菜", "updateTime": 1716540000000, "...": "..." }
    ],
    "collection": [],
    "memo": [
      { "id": 7, "content": "明天开会", "updateTime": 1716530000000, "...": "..." }
    ]
  }
}
```

**Push 请求体**（上传本地变更）：
```json
{
  "deviceId": "phone-a-xxx",
  "changes": {
    "task": {
      "upserted": [
        { "id": 99, "title": "新任务", "updateTime": 1716550000000, "...": "..." }
      ],
      "deleted": [42]
    },
    "collection": {
      "upserted": [],
      "deleted": []
    }
  }
}
```

**Push 响应**：返回服务端最终状态，检测冲突：
```json
{
  "conflicts": {
    "task": [
      {
        "local":  { "id": 42, "title": "买菜v2", "updateTime": 1716540000000 },
        "server": { "id": 42, "title": "买菜",   "updateTime": 1716550000000 }
      }
    ]
  },
  "accepted": 5,
  "rejected": 1
}
```

#### 8.5 安全设计

| 层级 | 措施 | 说明 |
|------|------|------|
| **传输层** | HTTPS / TLS 1.3 | 自签证书 + Let's Encrypt 自动续签 |
| **认证层** | HMAC-SHA256 签名 | 请求体 + Timestamp + Nonce 签名，防重放（5分钟窗口） |
| **密钥存储** | 客户端本地加密存储 | 密钥存入系统 KeyStore，不落盘明文 |
| **数据隔离** | 密钥级数据分区 | 不同密钥的数据完全隔离，互不可见 |
| **速率限制** | Token Bucket | 每密钥 60 req/min，单 IP 120 req/min |

HMAC 签名算法：
```
Signature = HMAC-SHA256(
  key = SyncKey,
  message = Method + Path + Timestamp + Nonce + BodyHash
)
请求头: X-Sync-Key: <KeyId>
        X-Sync-Time: <UnixMilli>
        X-Sync-Nonce: <Random32Hex>
        X-Sync-Sign: <SignatureHex>
```

#### 8.6 同步流程

```
设备 A（离线改了数据）                服务器                  设备 B

   │ 手动/自动触发同步                  │                      │
   │──── POST /pull ────────────────→  │                      │
   │  (传入各表 lastSync)              │                      │
   │←── 返回增量数据 ────────────────  │                      │
   │                                   │                      │
   │  合并冲突（LWW: updateTime大的胜）│                      │
   │                                   │                      │
   │──── POST /push ────────────────→  │                      │
   │  (上传本地 upserted + deleted)    │                      │
   │←── 返回冲突列表 ────────────────  │                      │
   │                                   │                      │
   │  解决冲突后再次 push               │                      │
   │                                   │                      │
   │                                   │    定时 pull          │
   │                                   │←── POST /pull ───────│
   │                                   │──→ 返回增量 ─────────│
   │                                   │                      │
   │                                   │    本地合并           │
```

#### 8.7 客户端数据库表结构适配

每张表新增 2 个字段，用于增量同步追踪：

| 字段 | 类型 | 说明 |
|------|------|------|
| `deviceId` | TEXT | 写入设备唯一标识 |
| `syncStatus` | INTEGER | 0=未同步  1=已同步 |

- `updateTime`（已有字段）作为增量拉取的时间锚点
- `syncStatus` 标记本地变更是否已推送到服务器
- 删除操作标记 `isDeleted = 1` 软删除，`syncStatus = 0` 等待推送到服务端

#### 8.8 客户端配置项

用户可在设置页配置：

| 配置项 | 说明 | 示例 |
|--------|------|------|
| 服务器地址 | 自部署服务器 URL | `https://my-server.com:8443` |
| 同步密钥 | 32位Hex密钥（可扫码） | `dt_sk_a1b2c3...` |
| 设备名称 | 用于冲突日志展示 | `我的手机` |
| 自动同步 | 开启后定时后台同步 | 每 5 分钟 |
| 仅 WiFi 同步 | 节省流量 | 开 |

#### 8.9 服务器部署（一键 Docker）

```dockerfile
# Dockerfile
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=1 go build -o sync-server .

FROM alpine:3.19
RUN apk add --no-cache ca-certificates
COPY --from=builder /app/sync-server /app/sync-server
EXPOSE 8443
VOLUME /data
CMD ["/app/sync-server", "-port=8443", "-db=/data/sync.db"]
```

```bash
# 一行命令部署
docker run -d \
  --name dungeon-sync \
  -p 8443:8443 \
  -v /opt/dungeon-data:/data \
  -e DOMAIN=my-server.com \
  your-registry/dungeon-sync:latest
```
