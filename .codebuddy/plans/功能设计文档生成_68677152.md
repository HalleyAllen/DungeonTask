## 产品概述

基于 requirements.md，生成功能设计文档 design.md，为 DungeonTask 鸿蒙应用开发提供完整的技术方案和设计规范。

## 核心任务

1. 对 requirements.md 中的六模块需求进行系统分析，明确每个模块的功能边界和数据需求
2. 设计整体应用架构（分层架构、模块划分、目录结构）
3. 设计各模块数据模型（ArkTS 接口定义、RDB 表结构）
4. 设计存储方案（Preferences 与 RDB 的分工策略）
5. 设计页面结构与导航流转（底部 Tab 导航 + 各模块页面层级）
6. 设计数据导出功能（JSON 格式、文件写入方案）
7. 将所有设计内容整理输出为 design.md 文件

## 技术栈选型

- 开发平台：HarmonyOS SDK 6.0.2，ArkTS + ArkUI
- 包名：com.example.dungeontask（已有）
- 数据存储：@ohos.data.preferences（配置/简单数据） + @ohos.data.relationalStore（结构化数据 RDB）
- 文件导出：@ohos.file.fs 写入应用沙箱，@ohos.file.picker 让用户选择保存位置
- 网络：本次不涉及

## 实现方案

### 总体架构

采用经典分层架构，职责清晰、便于维护：

```
表现层（Pages）
  ├── 主页（底部 Tab 导航）
  ├── 任务模块页面
  ├── 健身模块页面
  ├── 句子模块页面
  ├── 词语模块页面
  ├── 目标模块页面
  └── 备忘录模块页面
业务逻辑层（Service）
  ├── TaskService
  ├── FitnessService
  ├── SentenceService
  ├── WordService
  ├── GoalService
  └── MemoService
数据访问层（Repository / DB）
  ├── DbHelper（RDB 初始化、建表）
  └── PreferencesHelper（轻量配置存储）
工具层（Utils）
  ├── ExportUtil（数据导出 JSON）
  └── DateUtil（日期格式化）
```

### 存储方案设计

- **Preferences** 存储：应用设置、各模块列表页的排序/筛选偏好
- **RDB 关系型数据库** 存储：所有模块的核心业务数据（任务、健身记录、句子、词语、目标、备忘录）
- 统一在 DbHelper 中完成建表（version 管理），各 Service 通过 SQL 操作数据

### 页面导航设计

- 入口页面：Index（改为底部 Tab 主页）
- 底部 Tab 导航共 6 个 Tab，对应 6 个模块
- 各模块内部：列表页 → 详情/编辑页（pushUrl 跳转）
- 导出功能：各模块列表页右上角菜单触发

### 数据导出设计

- 格式：JSON（结构清晰，便于后续导入）
- 方式：通过 @ohos.file.picker 拉起系统文件选择器，用户选择保存目录
- 内容：按模块导出对应数据表中的所有记录

## 关键数据结构（ArkTS 接口）

### 任务 Task

```typescript
interface Task {
  id: number;           // 自增主键
  title: string;        // 任务标题
  content: string;      // 任务内容（可选）
  date: string;         // 所属日期 YYYY-MM-DD
  completed: number;    // 是否完成 0/1
  createTime: number;   // 创建时间戳
}
```

### 健身计划 FitnessPlan + 健身打卡 FitnessCheckin

```typescript
interface FitnessPlan {
  id: number;
  name: string;         // 计划名称，如"周一胸肌训练"
  weekDay: number;      // 星期几 1-7
  exercises: string;    // 训练项目（JSON 字符串）
  createTime: number;
}

interface FitnessCheckin {
  id: number;
  planId: number;       // 关联 FitnessPlan
  date: string;         // 打卡日期 YYYY-MM-DD
  completed: number;    // 是否完成 0/1
}
```

### 句子 Sentence

```typescript
interface Sentence {
  id: number;
  content: string;      // 句子内容
  source: string;       // 来源（可选）
  tag: string;          // 标签（可选）
  createTime: number;
}
```

### 词语 Word

```typescript
interface Word {
  id: number;
  name: string;         // 词语名称
  pinyin: string;       // 拼音
  explanation: string;  // 释义
  createTime: number;
}
```

### 目标 Goal

```typescript
interface Goal {
  id: number;
  title: string;        // 目标标题
  type: number;         // 类型：1=短期 2=长期
  progress: number;     // 进度 0-100（百分比）
  completed: number;    // 是否完成 0/1
  createTime: number;
}
```

### 备忘录 Memo

```typescript
interface Memo {
  id: number;
  content: string;      // 备忘录内容
  createTime: number;   // 创建时间戳（用于倒序排序）
  updateTime: number;    // 更新时间戳
}
```

## 实施要点

- 所有 Service 类使用单例模式，确保数据库连接的统一管理和资源释放
- RDB 表名与 Service 类名保持对应关系，便于维护
- 导出 JSON 结构按模块分 key，便于后续解析和导入
- 页面跳转使用 router.pushUrl，参数通过 params 传递 id
